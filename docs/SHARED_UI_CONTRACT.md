# TrailNote 共享 UI 契约（多端统一）

## 1. 契约目的

本契约用于约束 iOS、Android、H5、微信小程序四端 UI 与交互一致性。  
设计真源固定为：

- `prototype/tokens.json`
- `prototype/theme.css`
- `prototype/` 交互原型

任何端侧实现出现差异时，先以真源为准修正。

## 2. 视觉样式约定

### 2.1 颜色

- 品牌主色：`#285443`，强调色：`#17352C`、`#3D7A61`
- 辅助色：Lake `#3A7CA5`、Sand `#C59D5F`、Clay `#9B6B4F`
- 语义色：
  - 成功 `#2D8A57`
  - 警告 `#C88519`
  - 错误 `#BE4D3F`
  - 信息 `#3A7CA5`
- 页面背景使用浅色面：`#F4F6F3` / `#FFFFFF`，避免深色大面积默认主题漂移。

### 2.2 排版

- 正文字体统一以中文可读性优先（与 `theme.css` 的 `--font-body` 对齐）
- 字号层级对齐 token：`32/28/24/20/18/16/14/12`
- 关键数字信息（距离、时长、海拔）需使用更高字重（`500/600`）

### 2.3 圆角、阴影、间距

- 圆角：
  - 小控件：10
  - 常规组件：14/18
  - 卡片/抽屉：24+
- 阴影：
  - 卡片：`shadow-soft`
  - 悬浮层：`shadow-float`
- 间距基线使用 4 的倍数，优先采用 token：`4/8/10/12/14/16/18/20/24/32`

### 2.4 组件约定

### 按钮

- 主按钮高度 52，次按钮高度 44
- 禁止在同一页面混用多个主色按钮样式
- 禁用态必须降对比并禁点击，不得仅做文案提示

### 卡片

- 统一白色面板 + 细边框 + 轻阴影
- 卡片信息顺序固定：标题 -> 标签/难度 -> 指标 -> 操作

### 列表

- 列表项点击态、按压态、骨架态必须存在
- 业务端列表统一采用“首屏加载 + 下拉刷新 + 底部增量加载”模式，不在消费端页面使用分页按钮

### 底部导航

- 四主入口：发现 / 搜索 / 记录 / 我的
- “记录”入口在视觉权重上高于其他 tab
- 图标语义与顺序不可改动

### 页面顶部栏

- 返回按钮位置与行为一致（左上）
- 标题居中策略按平台习惯可微调，但文案与层级一致
- 右侧操作位只放当前页核心动作

### 弹层（Modal / Sheet）

- 所有关键确认操作使用二次确认弹层（如结束记录、状态流转）
- 弹层必须包含标题、说明、主按钮、取消按钮

### Toast

- 轻反馈统一 toast，时长约 2s
- 不用于承载复杂错误详情

### 空态

- 空态必须包含：状态标题 + 说明 + 引导动作
- 空态文案避免“暂无数据”孤立出现

### 加载态

- 首屏加载显示骨架屏或 loading 占位，不得白屏等待
- 局部操作使用局部 loading，不阻塞整页

### 表单校验

- 统一 `blur` 触发 + 提交兜底校验
- 错误文案使用短句，直接指出修复方式
- 必填项标识与错误色统一

### 记录链路状态样式

- `idle`：中性色，强调“准备开始”
- `recording`：品牌主色高亮 + 动态提示
- `paused`：警告色提示 + 可恢复入口
- `ended`：完成态提示 + 下一步入口（进入整理/创作）

## 3. 交互统一规则

- 页面返回：优先返回上一页；若无历史栈则返回模块首页，不可直接退出应用。
- 列表刷新：下拉刷新恢复到最新数据，刷新完成必须给出可见反馈。
- 收藏/关注切换：采用乐观更新，失败回滚并提示“操作失败，请重试”。
- 评论发布：提交中禁重复点击，成功清空输入并刷新最新评论。
- 草稿保存：支持手动保存 + 关键字段变更后自动保存（若端能力允许）。
- 状态提示：成功用轻提示，失败给明确原因，处理中给显式 loading。
- 错误反馈：接口错误优先展示可执行动作（重试/返回/联系客服）。
- 禁用态：视觉显式弱化 + 行为禁用 + 原因文案（如审核中、权限不足）。
- 弱网态：展示弱网提示条；关键操作允许重试。仅记录链路与草稿保存允许进入本地离线队列，评论发布不进入 SQLite 离线排队。

## 4. 平台差异边界

### 4.1 允许差异

- 系统返回手势与返回动效可遵循平台原生习惯
- 系统组件外观细节（输入框光标、键盘样式、状态栏高度）可按平台适配
- 动画曲线和时长可在不改变语义前提下做小幅调优

### 4.2 禁止差异

- 禁止改变信息架构与主导航顺序
- 禁止修改核心状态命名与状态流转语义
- 禁止出现同一业务操作在不同端使用不同字段含义
- 禁止擅自新增“仅单端存在”的关键流程分支（如额外审核步骤）
- 禁止脱离 `prototype/tokens.json` 自定义一套主视觉体系

## 5. 实施要求

- 新端开发前先完成 token 映射表（CSS -> SwiftUI -> Compose -> 小程序样式变量）
- 每个页面提测需附“契约对照检查项”
- 若确需偏离契约，必须先在 `docs/` 提交差异说明并评审通过
