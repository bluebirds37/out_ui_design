# TrailNote 正式开发总规划

## 1. 当前结论

当前项目已完成：

- 产品与 UI 设计规范
- 静态设计图
- Web 移动端高保真原型
- Design Token 与共享主题样式

当前项目尚未完成：

- iOS 客户端工程
- Android 客户端工程
- H5 客户端正式工程
- 微信小程序工程
- Spring Boot 后端
- MyBatis-Plus 数据层
- 移动端本地 SQLite 数据能力
- 后台管理后端
- 后台管理前端代码接入与适配

## 2. 关键现实判断

### 2.1 目前缺失的重要输入

当前仓库中没有发现以下内容：

- 你提到的后台管理前端代码
- 已有后端工程骨架
- 现成数据库设计
- 地图服务选型
- 对象存储选型
- 鉴权方案说明

因此正式开发前必须先补齐或确认这些前置条件，否则无法直接高质量进入全栈实现阶段。

### 2.2 多端同步开发的正确策略

不能直接“一次性并行写四端 UI + 双后端 + 管理系统”，否则极易出现：

- 交互不一致
- 数据结构反复变更
- 多端重复返工
- 后台接口多次推翻

正确做法是：

1. 先冻结统一设计语言与交互规则
2. 先定义领域模型与接口契约
3. 先完成一套主实现基线
4. 再将其同步扩展到多端

## 3. 建议总体技术路线

### 3.1 客户端建议

为了保证 UI 高度统一，同时兼顾 iOS、Android、H5、小程序，建议采用以下策略：

#### 方案 A：统一前端技术栈优先

- H5：`Vue 3 + Vite + TypeScript`
- 微信小程序：`uni-app x` 或 `Taro`
- iOS / Android：优先 `uni-app` 或 `Flutter`

优点：

- UI 更易统一
- 组件可复用
- 状态与业务逻辑更易共享

风险：

- 若追求非常强的原生体验，需要在局部做平台适配

#### 方案 B：原生体验优先

- iOS：SwiftUI
- Android：Jetpack Compose
- H5：Vue 3
- 小程序：原生小程序 / Taro

优点：

- 原生体验更强

风险：

- 成本高很多
- UI 一致性维护难度大
- 需求变更时四端联动成本极高

### 3.2 结合你当前目标的推荐

推荐优先采用：

- 多端统一前端框架路线

推荐组合：

- 客户端统一：`uni-app + Vue 3 + TypeScript`
- H5：同一套工程直接支持
- 微信小程序：同一套工程编译
- iOS / Android：同一套工程编译

说明：

- 这最符合你“几个客户端严格把握 UI 统一大关”的要求
- 当前已有 Web 原型，转向 Vue 系多端工程会更自然

## 4. 后端总体架构建议

### 4.1 服务拆分

建议拆成两类后端：

#### A. 业务主后端

- Spring Boot
- MyBatis-Plus
- MySQL / PostgreSQL 作为主业务库
- Redis 作为缓存与会话支持
- MinIO / OSS / COS 作为媒体资源存储

职责：

- 用户
- 路线
- 点位
- 评论
- 收藏
- 媒体信息
- 发布与审核
- 搜索与筛选

#### B. 后台管理后端

说明：

- 可以是同一个 Spring Boot 多模块
- 也可以是独立的 admin-api 模块

职责：

- 后台登录
- 权限
- 用户管理
- 内容审核
- 路线管理
- 点位与媒体管理
- 举报处理
- 运营数据统计

### 4.2 SQLite 的正确定位

SQLite 不建议作为正式“服务端后端数据库”主库。

更合理的定位是：

- 客户端本地离线数据层

适合存：

- 记录中轨迹草稿
- 未上传点位草稿
- 离线媒体上传任务
- 页面缓存

所以：

- 主后端数据库：MySQL / PostgreSQL
- 移动端本地数据库：SQLite

```text

  common/
```

```text

```

## 6. 统一设计落地策略

### 6.1 单一设计源

当前统一设计源应以以下文件为准：

- `prototype/tokens.json`
- `prototype/theme.css`
- `HikeShare_UI_Design_Spec.md`
- `ui/export/` 中的设计图

### 6.2 禁止出现的多端差异

- 页面层级结构不同
- 同一功能按钮位置差异过大
- 同一交互入口在不同端命名不同
- 同一业务状态在不同端反馈不一致

### 6.3 可接受的平台差异

- 返回手势
- 系统权限弹窗方式
- 原生导航栏细节
- 键盘适配细节

## 8. 记忆与持续进度存储方案

### 8.1 当前要求

你要求“开发过程中自动存储记忆，方便下次运行时候自动加载”。

### 8.2 建议本地记忆文件

建议固定使用以下本地文件：

- `PROJECT_PROGRESS.md`
- `DEVELOPMENT_MASTER_PLAN.md`
- `memory/DECISIONS.md`
- `memory/TODO.md`
- `memory/CHANGELOG.md`

规则：

- 每次阶段完成后更新
- 只写事实、决定、待办、风险
- 不写密钥、口令、Token

## 9. 开发规范要求

### 9.1 通用

- 严格按语言与框架规范编码
- 增加必要注释，尤其是状态机、复杂交互、业务边界
- 目录结构按模块职责明确拆分
- 避免单文件过大

### 9.2 后端

- Controller 只做入参出参与鉴权入口
- Service 做业务逻辑
- Mapper 做数据访问
- DTO / VO / Entity 分离
- 统一异常处理
- 统一响应结构

### 9.3 前端

- 页面、组件、状态、接口分层
- 组件命名语义化
- token 驱动样式
- 避免把业务状态散落在样式逻辑里

## 10. 自动调试与测试策略

### 10.1 客户端

- 页面渲染测试
- 关键流程点击测试
- 表单校验测试
- 状态流转测试

### 10.2 后端

- 单元测试
- 集成测试
- Mapper 测试
- Controller 接口测试

### 10.3 原则

- 新功能必须附带最基本验证
- 关键流程必须能本地跑通

## 11. 当前最重要的前置阻塞

在正式进入全量开发前，必须确认以下事项：

1. 后台管理前端代码在哪里
2. 是否接受统一多端框架方案
3. 主数据库选 MySQL 还是 PostgreSQL
4. 地图服务选型
5. 对象存储选型
6. 登录方式范围

## 12. 下一步建议

下一步不要直接同时写全部端。

建议按下面顺序推进：

1. 先确认技术选型与缺失输入
2. 搭建正式工程骨架
3. 建立 memory 目录与持续记录机制
4. 先做 Spring Boot 后端底座与统一客户端主工程
5. 再同步展开 H5 / 小程序 / 后台管理

## 13. 当前建议的执行决策

如无额外约束，建议默认采用：

- 客户端：`uni-app + Vue 3 + TypeScript`
- 主后端：`Spring Boot + MyBatis-Plus + MySQL`
- 本地离线：`SQLite`
- 后台管理：基于你已有前端代码继续开发
- 记忆机制：本地 `memory/` 文件持续写入

这份规划文件用于后续正式开发的总基线。
