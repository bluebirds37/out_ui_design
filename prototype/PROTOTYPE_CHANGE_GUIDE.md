# TrailNote Web Prototype 需求变更规范

## 1. 目标

本规范用于约束后续所有基于 `prototype/` 目录的需求变更，确保每次迭代都能：

- 延续同一套设计 token 与视觉语言
- 在现有移动端 Web 原型上增量修改
- 保持交互逻辑、页面结构、组件行为的一致性
- 让需求、设计、前端修改路径清晰可追踪

## 2. 原型目录职责

### 2.1 文件职责

- `prototype/index.html`
  原型入口，不承载复杂业务逻辑。
- `prototype/theme.css`
  全局主题、组件基础样式、状态样式。
- `prototype/tokens.json`
  设计 token 源文件，颜色、间距、圆角、阴影、动效等统一从这里沉淀。
- `prototype/src/app.js`
  页面渲染、状态管理、交互流程、页面切换逻辑。

### 2.2 修改原则

- 视觉值优先先进入 `tokens.json` 与 `theme.css`
- 页面逻辑优先在 `app.js` 中做结构化扩展
- 不直接在单个页面中重复写新的视觉规则，除非是明确的一次性实验

## 3. 每次需求变更的标准输入

后续每次提需求，建议至少包含以下内容：

1. 变更目标
   这次想解决什么问题。
2. 影响范围
   会影响哪些页面、组件或流程。
3. 用户动作
   用户点哪里、看到什么、下一步是什么。
4. 状态变化
   默认态、加载态、成功态、失败态是否变化。
5. 是否影响 token / 组件
   如果会影响，需要同步更新设计系统层。

## 4. 需求描述模板

建议以后直接按下面格式提：

```md
需求名称：

变更目标：

影响页面：

影响组件：

交互变化：

新增状态：

是否影响 token：

验收标准：
```

## 5. 修改优先级

### P0

- 主流程可用性问题
- 页面点击后无反馈
- 状态流转错误
- 严重视觉错位影响理解

### P1

- 页面信息结构调整
- 表单与交互优化
- 新增业务模块
- 组件能力补全

### P2

- 动效优化
- 文案调整
- 局部视觉精修
- 展示性增强

## 6. 变更实施顺序

每次变更默认按这个顺序处理：

1. 明确影响页面与流程
2. 判断是否影响设计 token
3. 判断是否需要新增组件或状态
4. 修改 `app.js` 中的状态与交互逻辑
5. 修改 `theme.css` 中的组件与状态样式
6. 如有必要，更新 `tokens.json`
7. 本地验证点击流与状态结果

## 7. 设计与实现边界

### 7.1 可以直接改

- 页面布局
- 页面切换
- 弹层
- 表单行为
- 收藏逻辑
- 记录流程状态机
- 提示反馈

### 7.2 改之前应先确认

- 一级导航结构变化
- 核心品牌色变化
- 整体视觉方向变化
- 数据结构大改
- 原型是否要升级为框架项目

## 8. token 变更规则

### 8.1 必须进 token 的内容

- 品牌色
- 功能色
- 字体体系
- 间距尺度
- 圆角
- 阴影
- 动效 duration / easing
- 高复用尺寸

### 8.2 不应直接散落在页面里的内容

- 新的 raw hex 颜色
- 新的 magic number 间距
- 新的组件圆角值
- 高复用按钮尺寸

## 9. 页面新增规则

新增页面时，优先复用以下模式：

- `safe()` 页面容器
- `topbar()` 顶栏
- `bottomNav()` 底部导航
- `formField()` 表单字段
- `favoriteButton()` 收藏切换
- 现有 card / sheet / list-card / metric-card 样式

不要新增完全脱离现有系统的页面骨架，除非需求明确要求新视觉分支。

## 10. 交互新增规则

新增交互时优先走以下路径：

- 简单确认：复用 modal
- 短反馈：复用 toast
- 页面主流程：增加状态字段并在 `app.js` 中显式流转
- 表单校验：走 `touched + validate` 模式

避免：

- 在 DOM 上零散绑定难以维护的匿名逻辑
- 把业务状态藏在样式类名里
- 用纯视觉切换替代真实状态流转

## 11. 验收最低标准

每次改完至少满足：

- 页面能正常打开
- 关键点击可触发
- 有状态变化的控件状态正确
- 不破坏已有流程
- 若新增组件或规则，需要同步更新 token 或样式层

## 12. 后续演进建议

当原型复杂度继续上升时，建议升级为：

1. `src/screens/`
   页面拆分
2. `src/components/`
   组件拆分
3. `src/state/`
   状态管理拆分
4. `src/data/`
   mock 数据拆分
5. 逐步迁移到 Vite + React 或其他明确框架

在那之前，当前 `prototype/` 结构足够支撑中等规模的需求迭代。
